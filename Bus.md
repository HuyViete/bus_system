# Bus Module — Edge Computing Architecture

The C++ Bus simulator in `Bus/` models a fleet of edge devices that generate realistic telemetry with pre-computed events, anomalies, and distance calculations.

## Architecture

Each Bus is an edge device that owns:
- **SpeedSimulator** — state machine producing realistic speed, events, and anomalies
- **RouteGraph** — precomputed cumulative distances + snapped stop positions for distance computation
- **Sender** — persistent TCP connection to server data port (3000)
- **Receiver** — persistent TCP connection to server command port (4000)
- **GPS** — 3-second tick loop using route waypoints + SpeedSimulator + RouteGraph
- **Database** — optional local SQLite buffer per bus

## Precomputed Transit Graph

The transit graph is generated offline by `scripts/build_transit_graph.js` and stored as `Bus/transit_graph.json`. It contains per-route:
- **Cumulative distances**: array parallel to waypoints, giving meters from route start at each waypoint index
- **Snapped stops**: stations projected onto the route polyline with their `dist_along_route`

This enables O(1) distance lookups at runtime — no haversine calls needed.

### Regenerating the Graph

```bash
node scripts/build_transit_graph.js
```

The script reads `Bus/routes.json` + `Bus/stations.json` and outputs `Bus/transit_graph.json`.

## Edge Computing: Event-First Simulation

The bus generates events first, then adjusts speed to match — not the other way around.

### DrivingState Machine

```
CRUISING ──→ HARD_BRAKING ──→ ACCELERATING ──→ CRUISING
    │                                              ↑
    ├──→ SPEEDING ──→ DECELERATING ────────────────┘
    │                                              ↑
    └──→ DECELERATING (approach stop) ──→ STOPPED_AT_STATION ──→ ACCELERATING
```

States:
- **CRUISING**: 15–45 km/h with small noise
- **ACCELERATING**: +3 km/h per tick toward target
- **DECELERATING**: -4 km/h per tick toward target
- **HARD_BRAKING**: sudden drop of 25–40 km/h, emits `hard_brake` anomaly
- **SPEEDING**: 82–100 km/h for 3–8 ticks, emits `speeding` anomaly
- **STOPPED_AT_STATION**: speed = 0 for 8–30 seconds, emits `arrival`/`departure` events

## Edge-Computed Fields

### Events & Anomalies

| Computation | Output | Server Impact |
|---|---|---|
| Hard brake | `edge_anomalies: ["hard_brake"]` | Server skips re-detection |
| Speeding | `edge_anomalies: ["speeding"]` | Server skips re-detection |
| Stop arrival | `stop_event: "arrival"` | Server writes event + headway |
| Stop departure + dwell | `stop_event: "departure", dwell_seconds: N` | Server writes event + dwell |

### Distances

| Computation | Output | How It Works |
|---|---|---|
| Distance along route | `dist_along_route` | Lookup `cumulativeDist[wp_index]` from transit graph |
| Next stop ID | `next_stop_id` | First stop with `distAlongRoute > bus distAlongRoute` |
| Distance to next stop | `dist_to_next_stop` | `nextStop.distAlongRoute - bus.distAlongRoute` |
| Distance between any stops | Derived | `stopB.distAlongRoute - stopA.distAlongRoute` |

### What Stays on the Server

- **Headway / bunching** — requires cross-bus data
- **GPS loss** — edge can't detect its own absence
- **Off-route** — requires route geometry validation
- **Trip lifecycle** — central trip table
- **ETA** — requires historical speed profiles + dwell times

## Enriched Packet Format

```json
{
  "vehicle_id": "3001",
  "route": 3,
  "latitude": 10.7526,
  "longitude": 106.6694,
  "speed": 28.5,
  "heading": 185.3,
  "timestamp": 1747384800,
  "edge_anomalies": [],
  "dist_along_route": 4250.3,
  "next_stop_id": 383552890,
  "dist_to_next_stop": 320.7
}
```

With stop event:
```json
{
  "stop_event": "departure",
  "stop_event_id": 383552890,
  "dwell_seconds": 18.0
}
```

## Startup Flow

1. `main()` loads `routes.json` (340 routes)
2. `main()` loads `stations.json` (~5,500 stops)
3. `main()` loads `transit_graph.json` (precomputed route graphs)
4. Per-route stop zones are cached (stations within 200m of any route waypoint)
5. Each spawned Bus receives its route waypoints + nearby stop zones + route graph
6. SpeedSimulator initialized with stop zones for station detection
7. GPS initialized with route graph for distance computation

## Bus ID Scheme

`vehicle_id = route_id × 1000 + sequence`

Examples: route 3, bus 1 → 3001 | route 11, bus 2 → 11002

## Edge Filter

Packets are only sent when:
- First packet (register bus on server)
- Moved > 5m
- Speed changed > 3 km/h
- 30-second heartbeat (10 ticks)
- Anomaly detected
- Stop event occurred

## Threading Model

Per active bus: 3 threads (GPS loop, Sender worker, Receiver worker).
Plus temporary thread pools for parallel spawn/stop operations.

## Build & Run

```bash
# From project root
build.bat          # Compiles Bus/bus.exe
run.bat            # Starts the fleet manager REPL

# Regenerate transit graph (only when route/station data changes)
node scripts/build_transit_graph.js

# REPL commands
start -n 100              # Spawn 100 buses across all routes
start -r 3 -n 10          # Spawn 10 buses on route 3
list                      # Show active buses
stop -a                   # Stop all buses
exit                      # Shut down
```

## Server Compatibility

The server handles both packet formats:
- **New (edge-computed)**: `stop_event`, `edge_anomalies`, and distance fields present → server persists directly, only runs server-only detection (gps_loss, off_route, headway, bunching)
- **Legacy**: fields missing → server runs full detection pipeline (backward compatible)

Distance fields (`dist_along_route`, `next_stop_id`, `dist_to_next_stop`) are stored in the `gps_latest` table for ETA computation and live map display.
